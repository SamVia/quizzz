## 1. Complex Enterprise Network Overview

This lab simulates a complex enterprise network utilizing L2 networks, spanning tree, VLANs, DHCP, router redundancy, Internet connectivity, and NAT.

### Network Topology Specifications

* The topology includes an L2 network mesh with three VLANs: VLAN 1 (the native VLAN), VLAN 2, and VLAN 3.


* Two routers (R1 and R2) serve as default gateways for VLAN hosts, handling inter-VLAN traffic and providing WAN (Internet) connectivity.


* Default gateway redundancy between the routers must be provided using HSRP.


* Hosts must receive dynamic IP addresses via a DHCP server configured on the router. For simplicity, the DHCP service is not redundant and is provided by only one router.


* To enable full Internet connectivity, NAT capabilities must be configured on both routers.


* 
**Constraint:** The standard Linux Host in GNS3 does not support DHCP services; you must use the VPCS host instead.

## 2. Core Technologies & Configuration Hints

### IP Addressing & DHCP

* All IP addresses must belong to a private addressing space (e.g., 10.0.0.0/8).


* In the POLITO network, the DNS service is provided by specific IP addresses: **130.192.3.21** and **130.192.3.24**.


* To acquire a dynamic IP address on VPCS hosts, turn on the DHCP client service using the following terminal command:


```text
dhcp

```



### NAT Configuration (R1 and R2)

* NAT on Cisco devices requires designating interfaces connected to hosts as "inbound" and interfaces connecting to the Internet as "outbound".


* The router must be configured with an access list to identify the exact set of host IP addresses allowed to connect to the Internet.


* Because the router's WAN interface (Fa0/1) has only one IP address, NAT must be configured using **overloading** to share this address for all NATted traffic.



### Connecting to the Internet in GNS3

* Internet connectivity is achieved by attaching the GNS3 NAT service (represented by a cloud icon) to the Fa0/1 interface of the routers.


* GNS3 NAT operates independently of the NAT configured on the routers.


* 
**NAT Cascading Example:** A packet from host **10.0.0.1/24** heading to the Internet will have its source IP translated to the router's WAN interface (e.g., **192.168.122.1/24**) by the router's NAT. The GNS3 NAT will then translate that source IP again to the IP of the VM running GNS3 (e.g., **172.16.203.62/32**). Packets traversing the datacenter may encounter 3 or 4 cascading NATs.


* You must force the router to acquire its WAN IP address automatically because the IP address used by GNS3 is not known a priori. Use the following command on interface Fa0/1:


```text
ip address dhcp

```



### General Debugging Strategies

* 
**Do not configure the entire network at once.** If a mistake is made, a fully configured network severely hinders the ability to locate the problem.


* Verify step-by-step: check interface status via `show` commands or host connectivity via `ping` before moving to the next component.


* Example: If WAN connectivity is enabled on a router, verify the router can reach the Internet *before* testing if a host can reach the Internet.


* In case of persistent problems, create an alternative simplified topology in another GNS3 VM to isolate and test the misbehaving feature.


* Rule out DNS failures before assuming network failure. Ping an IP address first, then ping a hostname:


```text
ping 8.8.8.8
ping www.google.com

```

## 3. Practical Exercises: Verification & Troubleshooting

### Verification Questions

1. Verify that the elected root bridge corresponds to the expected one.


2. Verify that the topology of each spanning tree corresponds to expectations.


3. Ensure no links are completely prevented from transporting user traffic (a link blocked for one VLAN might be used by another).


4. Verify the routes present on all L3 devices (routers, hosts). Determine why automatic routes are present and how they were added.


5. Capture traffic exiting host H2 to verify the MAC addresses present in IP packets traveling to the Internet and vice versa.


6. Analyze HSRP packets generated by R1 and R2, focusing specifically on IP/MAC addresses.


7. Check the DHCP server to see which addresses were assigned to host clients.


8. Verify infrastructure operations by ensuring hosts can ping each other and connect to the Internet.



### Troubleshooting Questions

1. Turn off router R1 (disconnect the Fa0/0 interface; **do not power off the router** to avoid losing configuration), verify network behavior, re-activate R1, turn off R2, and verify again.


2. Turn off either one of the WAN interfaces of the routers and verify if the network still behaves correctly.

* 
**Step 4:** Define your private IP addressing scheme (e.g., 10.0.0.0/8 subnetted per VLAN).


* **Step 5:** Configure Routers R1 and R2. Apply `ip address dhcp` on the Fa0/1 (WAN) interfaces. Configure inter-VLAN routing (subinterfaces) on Fa0/0.


* 
**Step 6:** Configure HSRP on R1 and R2's subinterfaces to provide default gateway redundancy for the VLANs.


* 
**Step 7:** Turn on a DHCP server on **only one** of the routers to serve IPs to the hosts.


* **Step 8:** Configure NAT on R1 and R2. Mark internal subinterfaces as inbound, Fa0/1 as outbound, create an Access List matching your private IP scheme, and enable overloading.

**Step 9:** Issue the `dhcp` command on all VPCS hosts to acquire an IP.


* 
**Step 10:** Verify connectivity end-to-end (`ping 8.8.8.8`). Complete the Phase 1 Verification and Troubleshooting tasks outlined above.



### Phase 2: Advanced Configuration with Multilayer Switches

* 
**Step 2:** Add a specific route targeting the 8.8.8.0/24 network to force all traffic for that destination to exit specifically through router ML-2.


* 
**Step 3:** Create an additional VLAN specifically to transport this routing traffic between ML-1 and ML-2.


* 
**Step 4:** Verify hosts can successfully reach the 8.8.8.0/24 destination.


* **Step 5:** Detach the WAN link on ML-2. Verify that this correctly causes a network outage for some hosts (since dynamic routes are not utilized in this lab to correct the path).



### Phase 3: Advanced Configuration with Local and Campus-Wide VLANs

* **Step 1:** Expand the ML topology to mimic two buildings. Building 1 has ML-3 connected to ML-1. Building 2 has ML-4 connected to ML-2. ML-3 and ML-4 connect to each other..


* 
**Step 2:** Ensure ML-1 and ML-2 continue to provide default gateway HSRP for VLANs 1, 2, and 3 (spanning across the campus).

**Step 4:** Create a new "backbone VLAN" to transport traffic between the scattered default gateways across the multiple L3 devices.


* **Step 5:** Configure additional routing. You must configure routes to allow users in different VLANs to ping each other, and specifically allow VLAN 4 users to route out to the Internet.

### Phase 1: The Base Topology

**1. Configure VLANs and Ports (On SW-1, SW-2, SW-3, SW-4)**
You need to create the VLANs, assign host-facing ports to "access" mode, and switch-to-switch/router ports to "trunk" mode.

```text
enable
configure terminal
vlan 2
 name VLAN_2
vlan 3
 name VLAN_3
exit

! For a port connected to a host (e.g., H2 on VLAN 2):
interface fastEthernet 1/5
 switchport mode access
 switchport access vlan 2
 no shutdown

! For a port connected to another switch or router:
interface fastEthernet 1/1
 switchport trunk encapsulation dot1q
 switchport mode trunk
 no shutdown

```

**2. Configure Per-VLAN Spanning Tree (STP) (On Switches)**
You must explicitly tell certain switches to act as the root bridge for specific VLANs to optimize traffic flow and prevent a single switch from doing all the work.

```text
! On SW-1 (Make it root for VLAN 1)
spanning-tree vlan 1 root primary

! On SW-2 (Make it root for VLAN 2)
spanning-tree vlan 2 root primary

```

**3. Configure Router Subinterfaces (Router-on-a-Stick on R1 & R2)**
To route between VLANs, your routers need virtual subinterfaces on their internal-facing physical port (e.g., `Fa0/0`).

```text
interface fastEthernet 0/0
 no shutdown

! Create subinterface for VLAN 2
interface fastEthernet 0/0.2
 encapsulation dot1Q 2
 ip address 10.0.2.2 255.255.255.0
 no shutdown

```

**4. Configure HSRP for Gateway Redundancy (On R1 & R2 Subinterfaces)**
Hosts need a single Default Gateway IP (the Virtual IP). R1 and R2 will share this IP. One will be Active, the other Standby.

```text
! Inside the subinterface (e.g., fa0/0.2) on R1:
standby 2 ip 10.0.2.254
standby 2 priority 110  ! Higher priority makes it the Active router
standby 2 preempt       ! Allows it to take back over if it reboots

! Inside the subinterface on R2:
standby 2 ip 10.0.2.254
standby 2 priority 100  ! Lower priority makes it Standby
standby 2 preempt

```

**5. Configure the DHCP Server (On R1 only)**
Give the hosts IP addresses dynamically. Use the POLITO DNS servers specified in the document.

```text
! Exclude the router and HSRP IPs from being handed out to hosts
ip dhcp excluded-address 10.0.2.1 10.0.2.10
ip dhcp excluded-address 10.0.2.254

! Create the pool for VLAN 2
ip dhcp pool VLAN2_POOL
 network 10.0.2.0 255.255.255.0
 default-router 10.0.2.254       ! This is the HSRP Virtual IP
 dns-server 130.192.3.21 130.192.3.24

```

**6. Configure NAT & Internet Access (On R1 & R2)**
Set the WAN interface to get a DHCP address from GNS3, define inside/outside NAT zones, and create the overload rule.

```text
! 1. Set the WAN interface to DHCP
interface fastEthernet 0/1
 ip address dhcp
 ip nat outside
 no shutdown

! 2. Set the internal subinterfaces to NAT inside
interface fastEthernet 0/0.2
 ip nat inside

! 3. Create an Access Control List (ACL) to define what IPs are allowed out
access-list 1 permit 10.0.0.0 0.255.255.255

! 4. Enable NAT Overload (PAT)
ip nat inside source list 1 interface fastEthernet 0/1 overload

```

**7. Host Configuration (On VPCS nodes)**
Open the console for each VPCS host and request an IP.

```text
dhcp
ping 8.8.8.8

```

### Phase 2 & 3: Multilayer Switches and Advanced Routing

**1. Create Switched Virtual Interfaces (SVIs) (On ML-1, ML-2, etc.)**
When you upgrade from a standard router/switch combo to a Multilayer Switch, you move the default gateway IP from physical subinterfaces directly onto the VLANs themselves.

```text
! Enable L3 routing globally on the switch
ip routing

! Create the SVI for VLAN 2
interface vlan 2
 ip address 10.0.2.2 255.255.255.0
 no shutdown

! You apply HSRP here exactly as you did on the router subinterfaces
standby 2 ip 10.0.2.254
standby 2 priority 110
standby 2 preempt

```

**2. Create a "Backbone VLAN" for Inter-Switch Routing**
To pass traffic between ML-1, ML-2, ML-3, and ML-4, dedicate a VLAN (e.g., VLAN 99) just for backbone traffic.

```text
vlan 99
 name BACKBONE

interface vlan 99
 ip address 10.0.99.1 255.255.255.0
 no shutdown

```

**3. Configure Specific/Static Routing (On ML-1 / ML-2)**
The lab asks you to force traffic destined for `8.8.8.0/24` out of a specific device (e.g., ML-2).

```text
! On ML-1, tell it to send 8.8.8.0/24 traffic to ML-2's Backbone IP
ip route 8.8.8.0 255.255.255.0 10.0.99.2

! Ensure you also have a default route (0.0.0.0) pointing to the Internet (GNS3 NAT)
ip route 0.0.0.0 0.0.0.0 dhcp 

```

* **Packet Captures (MAC Addresses & HSRP):**
You cannot do this purely in the Cisco CLI. In GNS3, you must:
1. Right-click the link between H2 and its switch, and select **Start Capture** (this opens Wireshark).
2. Ping `8.8.8.8` from H2.
3. Look at the Wireshark capture: The Destination MAC address leaving the host will belong to the **HSRP Virtual MAC** (usually ending in `00:00:0c:07:ac:XX`), *not* the physical MAC of R1 or R2.
4. To analyze HSRP packets, filter Wireshark by typing `hsrp`. You will see R1 and R2 sending "Hello" packets to multicast address `224.0.0.2`.



### 2. How to Execute the Troubleshooting Checks (Section 2.6)

The lab asks you to intentionally break the network to prove your redundancy (HSRP) works.

* **Test Router Redundancy (Turn off R1):**
Do *not* power off the router device in GNS3, as it will wipe the running config. Instead, shut down the inside interface.
```text
! On R1:
configure terminal
interface fastEthernet 0/0
shutdown

```


*Action:* Go to a VPCS host and continually ping 8.8.8.8 (`ping 8.8.8.8 -t`). When you shut down R1, the ping should drop for a few seconds and then resume as R2 takes over the Active HSRP role. Once verified, run `no shutdown` on R1.
* **Test WAN Failure:**
```text
! On R1:
configure terminal
interface fastEthernet 0/1
shutdown

```


*Action:* Verify if the network still behaves correctly. (Since we are using basic HSRP without interface tracking in this lab, shutting down the WAN link on the active router might actually cause an outage because HSRP doesn't know the outside link went down! This is a classic lab "gotcha" to observe).

### 3. A Quick Note on Phase 3 Routing (Section 4)

For the final campus-wide setup, the lab explicitly notes that "dynamic routes are not part of this lab." Therefore, to allow Building 1 (ML-3) and Building 2 (ML-4) to talk to each other and the Internet, you **must** use static routes.

